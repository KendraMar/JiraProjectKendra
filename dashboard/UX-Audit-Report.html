Critical Findings

Critical Interaction Body-level click-to-close drawer dismisses panel during normal use

App.tsx — DrawerContentBody onClick (~line 2107)

Issue: The entire main content area has an onClick handler that closes the issue/epic drawer. Clicks from toolbar buttons, search, table rows, and other controls bubble up and trigger panel close.

Impact: Users accidentally lose their editing context when clicking Refresh, Search, or any interactive element while the panel is open. Keyboard users have no equivalent close mechanism. Unsaved work in the panel can feel "lost" even though draft state persists.

Recommendation: Replace body-level close with either (a) event.target === event.currentTarget check to only close on direct backdrop clicks, (b) a semi-transparent overlay/scrim behind the drawer, or (c) Escape key close with proper focus management. Add a "discard changes?" confirmation if hasChanges is true.

High Findings

High Accessibility Sprint tabs are not keyboard accessible

App.tsx — DroppableTab (~lines 376–418), SprintTabs (~468–479)

Issue: Sprint navigation uses <div onClick> with no role="tab", tabIndex, aria-selected, or keyboard event handlers (Enter/Space/Arrow keys).

Impact: Keyboard-only and assistive technology users cannot switch between sprints.

Recommendation: Use PatternFly Tabs/Tab components, or add role="tab", tabIndex, aria-selected, and roving focus with Arrow/Home/End key handling.

High Accessibility Issue key uses fake link instead of button

App.tsx — IssueRow (~lines 1277–1283)

Issue: The Jira number uses <a href="#"> with preventDefault. It looks like a link but performs an in-app action (opening the detail panel).

Impact: Screen readers announce it as a navigation link; keyboard behavior differs from expectations.

Recommendation: Use Button variant="link" or a native <button> styled as a link.

High Accessibility Description editor has no accessible name or role

App.tsx — IssueDetailPanel contentEditable (~lines 984–1003)

Issue: The contentEditable div has no role="textbox", aria-multiline, or aria-label.

Impact: Screen reader users cannot discover or identify the description editing area.

Recommendation: Add role="textbox", aria-multiline="true", and aria-label="Issue description".

High Accessibility Due date input has no label association

App.tsx — IssueDetailPanel (~lines 1033–1058)

Issue: The native <input type="date"> is not connected to its DescriptionListTerm label via id/htmlFor or aria-labelledby.

Impact: Assistive technology may not associate the label with the control.

Recommendation: Add id/htmlFor pairing or use PatternFly DatePicker.

High Interaction No error handling on initial data load

App.tsx — loadData (~lines 1735–1750)

Issue: try/finally with no catch. API failures leave the dashboard empty or showing stale static data with no user-facing error or retry.

Impact: Users see a blank or demo-data dashboard with no explanation when the API is down or credentials are wrong.

Recommendation: Add catch, set an error state, and show an Alert with retry button.

High Interaction No loading indicator when opening detail panel

App.tsx — handleModify (~lines 1817–1829)

Issue: fetchSingleIssue runs with no spinner or skeleton in the drawer while fresh data loads.

Impact: Users see stale fields briefly or assume the UI froze on slow networks.

Recommendation: Show a Spinner or skeleton in the panel until fetchSingleIssue resolves.

High Interaction Comments are local-only until Save — no indication to user

App.tsx — IssueDetailPanel addComment (~lines 797–835)

Issue: Clicking "Add" appends the comment to local draft state only. The comment is not sent to Jira until the user clicks Save. There is no visual indication of this.

Impact: Users believe the comment was posted; data loss if they close the panel without saving.

Recommendation: Either post comments via API immediately on Add (with loading/error states), or add an inline hint: "Comments are saved when you click Save."

High Interaction Drag-and-drop has no in-flight loading state

App.tsx — handleDragEnd (~lines 1895–1935)

Issue: Optimistic UI update happens immediately, but there is no per-row or global loading indicator during the moveToSprint API call.

Impact: Users may initiate additional drags before the first completes; concurrent operations could conflict.

Recommendation: Add a "moving..." state (row dimming or global flag) and disable drag until the API settles.

High Corner Case Silent API failures on sprint move, status transition, and comment add

jiraService.ts — moveToSprint, transitionIssue, addComment (~lines 773–816)

Issue: These API calls never check res.ok. HTTP 400/403/500 responses are silently ignored, and the UI shows success toasts.

Impact: Users believe operations succeeded when Jira rejected them. Board and Jira diverge.

Recommendation: Check res.ok after every mutation; throw on failure; surface error toasts.

High Corner Case Post-create/update sub-operations fail silently

jiraService.ts — createIssue/updateIssue Promise.allSettled (~lines 900, 998)

Issue: Sprint move, status transition, and comment creation run via Promise.allSettled after issue create/update. Failures are not inspected or surfaced.

Impact: Issue created but wrong sprint, wrong status, or missing comments — user told "success."

Recommendation: Inspect allSettled results and surface partial-failure warnings in the toast.

High Corner Case Static fallback data shown without explanation

jiraService.ts — fetchIssues/fetchEpics (~lines 649–660, 1048–1084)

Issue: On API error, services silently return hardcoded static data. No banner or indication that the user is seeing demo data.

Impact: Users may act on stale/fake data believing it's live.

Recommendation: Show a persistent warning banner when fallback data is active. Provide a retry option.

Opportunities

Opportunity Workflow Remove redundant "Modify" from kebab menu

Clicking the issue key already opens the edit panel. "Modify" in the kebab duplicates this. Consider keeping kebab for "Clone" (and future actions like Delete) and adding visual affordance that the key is the primary edit entry point.

Opportunity Workflow Combined sort + filter column menu

Filter and sort are separate icons per column header. A single column dropdown menu (like Jira or spreadsheets) combining both would reduce scanning and clicks for power users.

Opportunity Workflow Keyboard shortcuts for frequent actions

No shortcuts for refresh (R), search focus (/), close panel (Escape), or next/previous issue. Adding these would significantly speed up power users.

Opportunity Workflow Multi-select and bulk operations

No ability to select multiple stories for bulk sprint move, status change, or assignment. Row checkboxes + a bulk toolbar would reduce repetitive work for admins managing many issues.

Opportunity Workflow Epic create panel could use stepped wizard pattern

The epic creation form is a single long scroll. Collapsible sections (mirroring the story panel) or a PF Wizard would reduce initial overload for first-time users.

Opportunity Workflow Derive team members from API instead of hardcoded list

The allMembers array is hardcoded. Fetching from loaded issues/epics (or Jira API) would keep the assignee/reporter lists current as the team changes.

Opportunity PatternFly PF PageSection variant props could replace inline background styles

App.tsx — PageSection (~line 2180)

Using PageSection variant / isFilled instead of style={{ backgroundColor: "var(...)" }} would be more idiomatic.
